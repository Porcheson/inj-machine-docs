# 注塑机控制系统代码 — 读后感和建议

## 一、整体印象

这是一套工业注塑机 PLC 控制程序的完整工程，采用 IEC 61131-3 结构化文本 (Structured Text) 编写，面向 KEBA 平台（从项目中 KEBA 说明书 PDF 可印证）。代码总量约 50 个源文件，覆盖了注塑机全部核心动作子系统：合模/调模、注射（射出+保压+射退）、储料、座台、托模（含组合模式）、中子、温控，以及完整的 IO 管理、Modbus 通讯、面板管理和模式调度。**整体架构质量相当高**——代码注释极其详尽（中文），分层清晰，命名规范统一，具备工业级软件工程的项目管理水平。

## 二、架构亮点

### 2.1 单一主入口 + 分层调度

[PRG_MainControl.st](PRG_MainControl.st) 是整个工程的唯一主任务入口，按固定调用顺序完成：

```
IN[] 原始绑定 → 输入语义映射 → Modbus 输入绑定 → HMI 参数同步 
→ 面板调度 → Modbus 输出绑定 → 命令仲裁 → 温控处理 
→ 机器主控(模式路由+模块执行+状态汇总) → 输出语义映射 → OUT[] 原始下发
```

这条调用链每一步职责单一、边界清晰，非常易于理解和维护。

### 2.2 多级命令仲裁机制

[FB_CommandArbiter.st](FB_CommandArbiter.st) 将命令源显式分层：

- **基础命令快照** — 保留尚未显式分配的字段
- **PLC 数字量命令源** — 负责公共启动/停止/复位/急停 (X50-X53)
- **面板命令源** — 负责模式选择与手动点动

这种设计使得后续增加 HMI 运行命令、远程命令或调试命令时有明确的扩展点，不会打乱已有仲裁逻辑。

### 2.3 10ms / 100ms 双节拍机制

主任务以 1ms 基础周期运行，内部通过分频计数器生成：
- **10ms 工艺拍** — 驱动模式路由和自动流程
- **100ms 温控拍** — 驱动温度计算

所有执行模块仍然每拍 1ms 被调用（保证轴层状态更新及时），但部分逻辑只在命中特定节拍时刷新。这种设计平衡了实时性与计算负载。

### 2.4 一致的模块设计模式

每个执行模块（FB_Clamp, FB_Inject, FB_Nozzle, FB_MoldAdjust, FB_Meter, FB_Eject, FB_Core）都遵循统一模式：

1. 输入：Start/Stop/EStop/Reset + 模式号 + 工艺参数 + 现场传感器
2. 内部：状态机 (CASE OF) + 时间累计 (TON 100ms) + 内部 FB_HydAxis
3. 输出：Busy/Done/Alarm/dwAlarmID/uiActHint/uiActTime + 到位状态 + 工艺命令 (压力/速度/位置)

这种一致性使得新模块可以快速"复制模板"开发。

### 2.5 托模双模块设计

[FB_Eject.st](FB_Eject.st) 负责基础进退；[FB_EjectMode.st](FB_EjectMode.st) 在此基础上封装了定次、震动、停留三种组合模式，两者共用同一根轴但通过 [FB_MachineCommandComposer](FB_MachineCommandComposer.st) 做互斥选择。这种 **"基础块 + 模式块复用基础块"** 的设计既避免了代码重复，又防止了两块同时争抢输出轴。

### 2.6 面板型号抽象

[FB_PanelDispatcher.st](FB_PanelDispatcher.st) 通过枚举 `E_PanelModel` 统一调度 MK110/MK116/MK150/MK2030 四种硬件面板。每种面板只维护按键排序号定义（如 [FB_MK2030.st](FB_MK2030.st)），读键和写灯统一由公用层 `FB_PanelReader` / `FB_PanelLampEncoder` 循环处理。新增型号只需新建一个映射块，调度层自动适配。

### 2.7 HMI 参数安全机制

[FB_HMIParaInterface.st](FB_HMIParaInterface.st) 实现了机器在线参数与 HMI 编辑缓冲区的彻底解耦：
- 画面只能修改编辑缓冲区
- "应用到机器"命令经上升沿检测 + 机器忙碌互锁后才真正写入在线参数
- 提供读回、脏标记、同步计数等完整状态反馈

这个设计从根本上杜绝了"运行中误改配方导致撞模"的风险。

### 2.8 输出映射层的动作提示语义

[FB_PLCMachineOutputMapper.st](FB_PLCMachineOutputMapper.st) 不重复判断模块内部状态，而是通过 **动作提示号 (uiActHint)** 来判定当前应输出哪个 Y 点位。例如合模方向通过 Hint 18（卸压）、40/84-86（开模）、30-31/41/81-83（合模）精准区分，避免了输出层猜测业务语义的耦合。

### 2.9 结构体驱动的数据边界

所有模块间数据交换都使用 TYPE STRUCT 定义，包括 [MachineControlTypes.st](MachineControlTypes.st)，[AxisTypes.st](AxisTypes.st)，[ModbusTypes.st](ModbusTypes.st)，[PanelTypes.st](PanelTypes.st) 等。全局变量统一收口到 [GlobalVars.st](GlobalVars.st)，实例统一收口到 [Instances.st](Instances.st)。不存在散落在各处的裸全局变量。

### 2.10 温控仿真/真实双模设计

[FB_EK312.st](FB_EK312.st) 当前是控制/仿真一体化实现，内部模拟了 12 路温度按 100ms 拍递增/递减，支持人工加热组 R0-R3。注释明确标注"若后续接入真实温控器，可保留本接口不变，仅替换内部驱动实现"——这是一个实践性很好的过渡设计策略。

## 三、值得关注的问题与改进建议

### 3.1 轴配置硬编码

**问题**：所有执行模块内部都有硬编码的轴配置常量：

```pascal
stAxisConfigLocal: ST_HydAxisConfig := (
    nAxisID := 1,  // 每个模块不同
    rMaxCylinderStroke := 10000.0,  // 全部相同
    rMaxVelocity := 1000.0,
    rMaxPressure := 1000.0,
    rPosTolerance := 0.5,
    rVelTolerance := 0.5,
    rPresTolerance := 0.5
);
```

7 个模块 7 份几乎完全相同的轴配置，仅 `nAxisID` 不同。

**建议**：将轴能力边界 (MaxStroke/MaxVelocity/MaxPressure/Tolerance) 提取到机器参数结构或全局常量中，由 HMI 或配置文件设定。不同机器的液压缸行程、能力上限差异很大，硬编码意味着每换一个机型就要改 7 处代码。

### 3.2 模拟量反馈全部置零

**问题**：[FB_MachineController.st](FB_MachineController.st) 中所有电子尺、压力尺和轴实际位置/速度/压力反馈统一按 0 占位传入：

```pascal
udiClampElecRulerVal := UDINT#0,
rClampActualPosition := LREAL#0.0,
rClampActualVelocity := LREAL#0.0,
rClampActualPressure := LREAL#0.0,
```

当前这意味着所有按位置完成的段 (uiMode=0) 都不会真正完成（因为位置反馈永远为 0，永远达不到目标位置）。**这在真实机器上运行时会导致严重的功能缺失。**

**建议**：优先建立模拟量输入采集链路（至少电子尺位置反馈），然后在主控层统一接入。可以考虑新增 `FB_AnalogInputBinder` 与 `ST_MachineAnalogSensor` 结构，按轴分别绑定位置、压力采集通道。此问题严重性最高，建议作为首要任务。

### 3.3 自动流程缺少关键互锁

**问题**：[FB_AutoCycle.st](FB_AutoCycle.st) 的自动流程步骤切换条件过于宽松：

- 合模步骤只等 `bMoldClosed OR bClampDone`，没有校验合模是否真正到达目标位置。如果合模块因局部卡顿报 Done=false，自动流程会永远卡住。
- 座进步骤同理只等 `bNozzleIned OR bNozzleDone`。
- 托模进/退没有与开模到位做互锁（开模未到位就允许托模进，可能撞模具）。

**建议**：
1. 所有自动步骤增加**超时保护**（当前只在模块级别有时间限制，流程级别缺少）
2. 托模进步骤增加 `bMoldOpened` 前置条件检查
3. 射胶步骤增加 `bNozzleIned AND bMoldClosed` 前置条件检查

### 3.4 中子功能耦合在合模参数中

**问题**：[ClampTypes.st](ClampTypes.st) 中定义了 `aCoreFn[1..8]`、`aCoreInMode[1..8]`、`aCoreInStroke[1..8]` 等 6 个中子相关数组，中子参数散落在合模参数结构里。但从架构上看，中子应该是一个独立的执行模块（已有独立的 FB_Core 和 CoreTypes）。这种耦合意味着：
- 修改中子参数需要改动合模类型定义
- 中子数量扩展到 8 以上需要改合模结构

**建议**：将中子参数从中子合模参数中移出，放到独立的 `ST_CorePara` 扩展中，或创建 `ST_CoreArrayPara` 承载多路中子。合模块只需保留 `aCoreAllow` 互锁结果，具体的启动模式/行程/位置由主控层从独立的中子参数区读取。

### 3.5 调试段回退逻辑存在隐患

**问题**：多个模块（FB_Clamp, FB_Inject, FB_Eject, FB_Nozzle, FB_Core）中的调试段回退逻辑是不一致的：

- `FB_Clamp`：检查 `uiPres=0 AND uiSpd=0 AND udiPos=0` 三个字段
- `FB_Core`：只检查 `uiPres=0 AND uiSpd=0` 两个字段
- `FB_Nozzle` / `FB_Eject`：检查三个字段

如果调试段只配置了压力而不配置速度（例如纯粹靠压力控制的动作），会被误判为"未配置"而回退到正式第一段。

**建议**：统一调试段有效性判定逻辑。建议增加一个显式的 `bEnable` 标志位，或统一使用一个"是否已配置"的判断函数 `F_IsSegValid`，避免各模块各自实现判空逻辑。

### 3.6 报警码管理分散

**问题**：报警码 (dwAlarmID) 在各模块中以位掩码形式分散定义：

- DWORD#16#0001 在不同模块中含义不同（低压超时/调模进超时/射出超时/中子模式错误等）
- 轴层报警位与模块报警位可能冲突（都用 DWORD#16#0010 表示急停）

**建议**：建立统一的报警码枚举或常量表（如 `AlarmCodes.st`），按位域分配：
- bit 0-7：模块级报警（每个模块独立一组）
- bit 8-11：轴层保护报警
- bit 12-15：系统级报警
这样任何位置的报警码都能被无歧义地追溯到根源。

### 3.7 缺少吹气/清料/拉带/润滑等外围模块

**问题**：[FB_PLCMachineOutputMapper.st](FB_PLCMachineOutputMapper.st) 中大量将吹气(Y26/Y27)、清料、倒带(Y13)、拉带(Y15/Y24/Y25)、润滑等输出点显式清零，注释标注"当前架构尚未建立XX模块命令边界"。面板调度块中也同样对吹气灯、润滑灯、定位灯等做了"当前先跟随按键值"的临时处理。

这说明目前的工程是一个**核心功能已完备、外围辅助功能待补充**的状态。

**建议**：按优先级建立外围模块的实现计划：
1. **吹气模块** — 优先级高，直接影响产品质量
2. **清料模块** — 与储料模块紧密配合
3. **定位模块** — 用于转盘/滑板等特殊配置
4. **拉带系统** — 用于带嵌件产品
5. **润滑系统** — 设备维保相关

### 3.8 PanelTypes.st 枚举缺少语法逗号

**问题**：[PanelTypes.st:16-22](PanelTypes.st#L16-L22) 中枚举定义：

```pascal
ePanel_MK110,
ePanel_MK116,
ePanel_MK150    // ← 缺少逗号
ePanel_MK2030
```

`ePanel_MK150` 后面缺少逗号。IEC 61131-3 对枚举分隔符的要求取决于具体编译器实现——有些允许省略最后一个逗号，有些要求必须有。这可能在特定编译器版本下导致编译错误。

**建议**：补上缺失的逗号，确保跨编译器兼容。

### 3.9 储料模块方向固定为单向

**问题**：[FB_Meter.st:211](FB_Meter.st#L211) 中调用轴层时 `bDir := TRUE` 固定为单向：

```pascal
bDir := TRUE,  // 当前储料流程只有单向执行语义
```

虽然储料确实通常只有单向，但从液压轴的角度看，"储料方向"和"松退方向"可能对应不同的输出阀。如果后续需要支持储料回退（例如更换原料时排空料筒），当前接口无法支持。

**建议**：保留 `bDir` 接口但当前先维持 TRUE 不变，或在注释中明确"储料回退将使用射退模块代替"。

### 3.10 缺少单元测试或仿真测试框架

**问题**：整个工程没有任何测试代码。对于 PLC 程序，虽然传统上依赖现场调试，但在当前仿真温控已经嵌入代码的情况下，完全可以构建一套 PLC 仿真测试环境。

**建议**：
1. 利用 `FB_EK312` 的仿真模式，搭建 HIL (Hardware-in-the-Loop) 级别的自动测试
2. 至少为自动流程状态机 (`FB_AutoCycle`) 编写状态转换测试用例
3. 为命令组合器 (`FB_MachineCommandComposer`) 的各类模式切换场景编写测试

### 3.11 文档与代码一致性风险

**问题**：[GlobalVars.st:9-12](GlobalVars.st#L9-L12) 的维护规则要求任何改动都须同步更新 3 份 MD 文档。但实际上 [Code/文档/](.) 路径下的那 3 份文档（开发规范与注释约定.md、总体架构与代码分析.md、修改记录.md）在当前目录中不存在——Glob 能找到它们但 Read 工具无法定位（可能是路径中包含中文字符导致跨平台兼容性问题）。

**建议**：将文档文件名中的中文字符改为英文或拼音，例如 `dev-standards.md`、`architecture.md`、`changelog.md`，避免跨工具链访问问题。

| 维度 | 评价 | 说明 |

| 架构设计 | 优秀 | 分层清晰、职责单一、扩展性强 |
| 命名规范 | 优秀 | 匈牙利命名 + 语义前缀 (FB_/ST_/E_/gvl_)，一致性好 |
| 注释质量 | 优秀 | 中文注释详尽、维护要求明确、每个块有功能说明 |

## 五、优先级建议

### 高优先级（影响真实运行）
1. 建立模拟量采集链路，接入电子尺位置反馈和压力反馈
2. 自动流程增加步骤级超时保护和关键互锁（托模前进需开模到位）
3. 将轴能力参数从硬编码提取为可配置项

### 中优先级（提升工程质量）
4. 建立统一报警码管理体系
5. 统一调试段有效性判定逻辑
6. 将中子参数从合模类型中解耦
7. 补全 PanelTypes.st 中的枚举逗号

### 低优先级（功能扩展与长期建设）
8. 按计划逐步实现吹气、清料、拉带、定位、润滑模块
9. 搭建仿真测试框架
10. 文档文件名去中文化以提高跨平台兼容性
