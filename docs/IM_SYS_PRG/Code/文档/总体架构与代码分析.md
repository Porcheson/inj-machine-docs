# 总体架构与代码分析

## 文档维护规则

自 `2026-04-13` 起，本文件与 `开发规范与注释约定.md`、`修改记录.md` 必须同步维护。凡是模块职责、主入口、接口边界、状态机流程发生变化，本文件都要同步更新。

## 当前统一架构

### 顶层执行链

`PRG_MainControl`

1. 先调用 `gvl_fbHMIParaInterface`，处理在线参数与 HMI 编辑缓冲区的同步。
2. 再在顶层生成固定 10ms 工艺节拍与固定 100ms 温控节拍，供后续主控块与温控块直接消费。
3. 再调用顶层温控实例 `gvl_fbEK312` 和机器主控实例 `gvl_fbMachineController`，统一完成温控刷新、模式判定、自动流程调度、子模块执行和状态汇总。
4. `PRG_MainControl` 绑定 1ms 基础任务；主控工艺逻辑由顶层 10ms 节拍驱动；温控计算由顶层 100ms 节拍驱动。
5. 顶层入口和机器主控的当前维护基线已提升为“逐变量详细注释 + 逐参数注释 + 逐执行行可直接读懂”的注释密度。

### 主控内部分层

`FB_MachineController`

1. 读取 `ST_MachineCommand`、`ST_MachinePara`、`ST_MachineSensor`。
2. 内部调用：
   `FB_ManualMode`、`FB_MoldAdjustMode`、`FB_AutoCycle`、`FB_Clamp`、`FB_MoldAdjust`、
   `FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core`。
3. 工艺路由、模式切换和自动流程只消费顶层送入的 10ms 节拍；执行模块与其内部轴层仍随 1ms 基础任务运行。
4. 8 个执行模块各自在模块内部持有 `FB_HydAxis`，主控只做工艺层命令分发和模块状态汇总。
5. 温控系统由顶层独立运行并只消费顶层送入的 100ms 节拍；`FB_MachineController` 不再保留 `stSensor.bTempReady` 这类温控前提接口，也不处理温控块接口和温控状态。
6. `PRG_MainControl.st` 与 `FB_MachineController.st` 现作为注释密度的新基线，要求变量定义和关键执行行保持详细注释。

### 全局边界

1. `GlobalVars.st` 只保留在线运行必需的共享结构。
2. `Instances.st` 只保留 `gvl_fbHMIParaInterface`、`gvl_fbEK312`、`gvl_fbMachineController` 三个顶层实例。
3. 旧版手动/自动中间缓存已经移除，避免旧架构与新架构同时生效。
4. 当前工程中的全局共享对象前缀已统一收口为 `gvl_`。
5. `gvl_stPanelButtons` 作为综合面板按键边界，供不同面板型号功能块与上层逻辑共享同一套按键/按灯状态。

## 模块职责分析

### HMI 系统

文件：`Code/HMI系统/FB_HMIParaInterface.st`

职责：

1. 管理在线参数与 HMI 编辑缓冲区之间的读回、下发和脏标记。
2. 在机器忙碌时阻止 HMI 直接下发在线工艺参数。
3. 为 HMI 提供统一状态反馈。

### 主程序入口

文件：`Code/主程序入口/*.st`

职责：

1. `PRG_MainControl` 是唯一有效主入口。
2. `MachineControlTypes.st` 定义机器级统一接口。
3. `FB_MachineController.st` 是整机唯一主控编排器。
4. `PRG_MainControl.st` 直接调用顶层 `gvl_fbEK312` 并生成 10ms/100ms 节拍，`FB_MachineController.st` 只消费顶层 10ms 节拍，不再驱动、消费或汇总温控块本体状态。

### 面板管理

文件：`Code/面板管理/*`

职责：

1. `PanelTypes.st` 定义 `ST_PanelButton` 与 `ST_PanelButtons`，统一收口面板排序号、键值和灯值。
2. `FB_MK110`、`FB_MK116`、`FB_MK150` 分别封装三个面板型号的排序号定义和输入解码逻辑。
3. 面板灯输出统一通过 `FB_PanelLampWriter` 按排序号写入 `wOUT0/wOUT1`，保证同一逻辑键的键值和灯值始终绑定到同一个排序号。
4. 面板型号差异只体现在各型号功能块中的 `uiPanelOrder` 定义和输入排序号读取函数，不再重复维护两套分离的键值/灯值映射。

### 模式管理

文件：`Code/模式管理/*`

职责：

1. `FB_ManualMode`：把机器手动命令路由到各执行模块。
2. `FB_MoldAdjustMode`：把调模模式下允许的动作收口，并强制锁模进入调试方向。
3. `FB_AutoCycle`：只做自动流程步骤调度，不直接驱动轴；自动前置条件当前只检查门禁、液压和系统无报警。
4. `PRG_*` 文件现为兼容占位，不再承担实际执行。

### 伺服系统

文件：`Code/伺服系统/液压轴/*`

职责：

1. `FB_HydAxis`：对模块写入的 `ST_AxisRefHyd` 做最终轴实例化。
2. 统一完成轴号绑定、能力上限回写、命令限幅和轴层报警叠加。
3. 当前 8 个执行模块都在模块内部各自实例化一份 `FB_HydAxis`：
   合模、调模、座台、射胶、储料、托模基础、托模模式、中子。
4. `AxisTypes.st` 新增了 `ST_HydAxisConfig`，用于各子模块内部统一收口轴号和保护边界。

### 合模系统

文件：`Code/合模系统/*`

职责：

1. `FB_Clamp`：处理开模、合模、低压、高压、调试等动作。
2. `FB_MoldAdjust`：处理模厚调节进退动作。
3. 对应 `*Types.st` 文件维护参数结构与状态枚举。

### 注射系统

文件：`Code/注射系统/*`

职责：

1. `FB_Nozzle`：座台进退控制。
2. `FB_Inject`：射出、保压、射退控制。
3. `FB_Meter`：储料动作控制。
4. 以上 3 个执行块已在 `2026-04-14` 补齐局部变量语义和状态机段落注释，后续修改时应保持同等注释粒度。

### 顶出系统

文件：`Code/顶出系统/*`

职责：

1. `FB_Eject`：基础托模进、保压、退动作。
2. `FB_EjectMode`：定次、震动、停留等组合托模模式。
3. `FB_Core`：中子进退动作。
4. `FB_EjectMode` 通过内部 `FB_Eject` 实例直接组合更高层托模动作，不再额外重复执行一层托模轴层。
5. 以上 3 个执行块已在 `2026-04-14` 补齐局部变量语义和状态机段落注释，后续修改时应保持同等注释粒度。

### 温度系统

文件：`Code/温度系统/EK312/*`

职责：

1. `FB_EK312`：12 路温控的仿真/控制接口。
2. 由 `PRG_MainControl` 顶层直接调用，独立刷新每路温控状态。
3. 温控块实例放在顶层实例区，由主入口直接驱动，不再下沉到机器主控内部，也不再把温控块状态并入机器主控。

## 本次统一后的关键结论

### 完整性

1. 类型定义、功能块定义和引用关系已做静态核对，当前 `Code` 目录内没有发现缺失的自定义类型或功能块引用。
2. 顶层架构已统一为“单入口 + 单主控”。
3. 新增了独立轴功能块层，执行模块与最终轴实例化已经解耦。
4. 旧版多入口残留逻辑已收敛为兼容占位，避免重复驱动。
5. 全局变量、全局实例命名已统一为 `gvl_` 前缀，顶层边界命名更明确。

### 可执行性

1. 已修复三处明确的 ST 语法问题：`FB_Clamp.st`、`FB_MachineController.st`、`FB_EK312.st` 中的 `END_FOR` 已补齐分号。
2. `PRG_MainControl` 现在已包含 HMI 同步和机器主控调用，单独绑定即可形成完整执行链。
3. `FB_Clamp`、`FB_MoldAdjust`、`FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core` 使用到的轴，已经下沉到各执行模块内部实例化，主控不再处理轴配置、轴使能和轴状态并入。
4. `FB_MachineController` 已改为消费 `PRG_MainControl` 顶层生成的 10ms 逻辑节拍；在 1ms 基础任务下，轴层和执行块可保持高频刷新，自动流程和模式路由按 10ms 刷新。
5. `FB_EK312` 已提升到 `PRG_MainControl` 顶层直接调用，并由顶层 100ms 节拍驱动刷新；`FB_MachineController` 不再消费或汇总温控块结果。
6. `FB_AutoCycle.st` 已补回本地 `VAR` 状态缓存声明，自动流程块恢复完整的功能块声明结构。
7. `FB_EK312.st` 在停机态和空闲态会主动清零加热输出，且只消费顶层下发的 100ms 刷新节拍。
8. 主干 `.st` 文件已统一整理为 `UTF-8 with BOM`，文件头、接口注释和类型注释的显示一致性已收口。
9. 由于当前工作区不包含 PLC 工程任务配置文件，也没有实际 ST 编译器，本项目尚未做编译级验证；1ms 基础任务绑定仍需在工程环境中确认。
10. `FB_Clamp`、`FB_MoldAdjust`、`FB_Nozzle`、`FB_Inject`、`FB_Meter`、`FB_Eject`、`FB_EjectMode`、`FB_Core` 已补齐局部变量定义说明、公共分支意图和状态机分段注释，便于后续按状态排查动作异常。
11. `PRG_MainControl` 与 `FB_MachineController` 已开始按“全部 `VAR_*` 变量必须详细注释、顶层调用参数逐行注释”的新规则收口，后续主干文件按同一标准继续补齐。
12. 温控块接口已进一步简化，`FB_EK312` 删除 `bManualMode` 与 `uiActTime`，主控侧同步删除温控镜像输入、局部缓存和机器级温控状态汇总。
13. 面板管理层已新增 `ST_PanelButtons` 综合结构以及 `FB_MK110`、`FB_MK116`、`FB_MK150` 三个面板功能块；面板型号切换时只需要调整对应逻辑键的 `uiPanelOrder` 定义。

### 架构风险

1. 如果 PLC 工程里仍把旧版 `PRG_MachineController`、`PRG_ManualMode`、`PRG_MoldAdjustMode`、`PRG_AutoCycle` 绑定到任务，会与新架构说明冲突。
2. 本次已把这些程序改为兼容占位，但工程层仍应检查任务映射，确保只运行 1ms 基础任务上的 `PRG_MainControl`。

## 后续维护建议

1. 新增模块时，先扩展 `MachineControlTypes.st` 和 `GlobalVars.st` 的统一边界，再扩展 `FB_MachineController`。
2. 修改任何状态机时，必须同步更新本文件中的职责说明和执行链描述。
3. 若后续接入真实 PLC 工程文件，建议补一份“任务配置与下载说明”文档，并在本文件追加编译验证记录。
